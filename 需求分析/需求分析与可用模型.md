## 二、产品核心定位

**一个 Web SSH 终端，AI 能力直接融合在命令行中。** 没有独立的聊天面板，没有模式切换。用户在同一个 shell 提示符下既可以输入 shell 命令，也可以输入自然语言，所有内容（终端输出、AI 回复、命令确认卡片）在同一个终端滚动流中内联呈现。

### 3.2 核心交互模式：统一命令行

**没有"Shell 模式"和"Agent 模式"的切换。** 只有一个终端界面，用户在 shell 提示符处输入：

- **输入 shell 命令** (如 `ls -la`, `rm -rf k8s`) → 直接通过 SSH 执行，输出显示在终端
- **输入自然语言** (如 `删除 k8s目录`, `查看并分析安装 https://xxx`) → AI 识别为自然语言，内联回复

系统需要自动判断用户输入是命令还是自然语言。

### 3.3 终端内联元素

所有内容在同一个终端滚动流中，按时间顺序从上往下排列：

#### A. 用户输入行
```
[admin@iZ0jo4lau3sirrhgl56g5mZ ~]$ 我有个 查看并分析 安装 https://github.com/obra/superpowers
```
- 显示在标准 shell 提示符后
- 自然语言中的关键词（URL、操作词）以蓝色/黄色高亮

#### B. AI 回复块
```
● 好的，我明白您想查看并分析安装来自 https://github.com/obra/superpowers 的软件。
  首先，我会尝试克隆该 GitHub 仓库到您的服务器上，以便查看和分析其内容。
```
- 蓝色圆点 `●` 作为 AI 回复的起始标记
- 支持 Markdown 渲染（链接、加粗、代码块、列表等）
- 回复结束后显示 `👍 👎 [开启新会话]`

#### C. 命令确认卡片（三种状态）

**状态1 - 已同意（自动执行的普通命令）：**
```
┌────────────────────────────────────────────────┐
│ ✓ 已同意                                       │
│                                                │
│   git clone https://github.com/obra/super...   │
└────────────────────────────────────────────────┘
```

**状态2 - 待确认（需要用户操作）：**
```
┌────────────────── [执行 Ctrl↵] [修改 CtrlE] [拒绝 CtrlQ] ──┐
│ 是否同意执行以下命令并查看输出？                              │
│                                                              │
│   cd superpowers && ls                                       │
└──────────────────────────────────────────────────────────────┘
```

**状态3 - 高危命令（执行按钮为红色 + 二次确认弹窗）：**
```
┌────────────────── [执行(红)] [修改 CtrlE] [拒绝 CtrlQ] ─────┐
│ 是否同意执行以下高危命令并查看输出？                          │
│                                                              │
│   sudo find / -name "kubernetes" 2>/dev/null                 │
└──────────────────────────────────────────────────────────────┘
         弹窗: "确认执行这条高危命令吗？" [取消] [确认]
```

#### D. 命令执行输出
```
[admin@iZ0jo4lau3sirrhgl56g5mZ ~]$ git clone https://github.com/obra/superpowers.git
Cloning into 'superpowers'...
remote: Enumerating objects: 4631, done.
...
Resolving deltas: 100% (1958/1958), done. 💡
[admin@iZ0jo4lau3sirrhgl56g5mZ ~]$
```
- 命令执行完毕后末尾显示 💡 灯泡图标
- 然后 AI 自动分析执行结果，继续内联回复

#### E. AI 思考状态
```
✨ AI 正在思考...
```
或
```
  正在生成命令...
```

### 3.4 完整交互流程

```
用户在 shell 提示符输入内容
         │
         ├─ 判断：是 shell 命令？
         │     ├─ 是 → 直接通过 SSH 执行，终端显示输出
         │     └─ 否 (自然语言) ↓
         │
    发送给 AI 分析
         │
    AI 内联回复 (● 标记)
         │
    AI 生成需要执行的命令
         │
    显示命令确认卡片
         ├─ 低危命令 → 自动设为"已同意"，直接执行
         ├─ 普通命令 → 显示 [执行] [修改] [拒绝] 按钮，等待用户操作
         └─ 高危命令 → [执行(红色)] + 二次确认弹窗
                │
         用户选择：
         ├─ 执行 → 通过 SSH 执行命令，终端显示输出，末尾 💡
         ├─ 修改 → 用户编辑命令内容后执行
         └─ 拒绝 → 跳过该命令
                │
    AI 分析执行结果 → 内联回复
         │
    如有后续步骤 → 继续生成命令卡片 (循环)
         │
    任务完成 → 输出总结
```

## 四、输入判断逻辑

系统需要区分用户在 shell 提示符处输入的是 **shell 命令** 还是 **自然语言**。判断策略：

| 判断依据 | shell 命令 | 自然语言 |
|---------|-----------|---------|
| 以已知命令开头 | `ls`, `cd`, `git`, `docker`, `cat`, `rm`, `sudo` ... | - |
| 包含中文字符 | - | `删除 k8s目录`, `查看内存` |
| 以路径开头 | `./script.sh`, `/usr/bin/xxx` | - |
| 包含管道/重定向 | `ls \| grep xxx`, `echo > file` | - |
| 纯英文但不像命令 | - | `check memory usage`, `install docker` |
| 混合自然语言+参数 | - | `我有个 查看并分析 安装 https://xxx` |

建议实现：先用简单规则匹配，无法确定时交给 AI 做意图分类。

### 5.3 前端关键设计：终端 + 内联 UI 混合渲染

这是最核心的技术难点。终端内容和 AI UI 元素在同一个滚动容器中交错排列。

**方案：虚拟终端 + 内联 DOM 注入**

```
终端滚动容器 (单一 div, overflow-y: auto)
  ├── [xterm 行] Welcome to Alibaba Cloud ...
  ├── [xterm 行] [admin@host ~]$ 用户输入内容
  ├── [React 组件] AI 回复块 { ● markdown内容 }
  ├── [React 组件] 命令确认卡片 { ✓ 已同意 / 待确认 }
  ├── [React 组件] 反馈按钮 { 👍 👎 开启新会话 }
  ├── [xterm 行] [admin@host ~]$ git clone xxx
  ├── [xterm 行] Cloning into 'xxx'...
  ├── [xterm 行] [admin@host ~]$
  ├── [React 组件] AI 回复块 { ● 分析结果 }
  ├── [React 组件] 命令确认卡片 { 待确认 }
  └── ...
```

实现思路：不使用纯 xterm.js 渲染所有内容，而是将 xterm.js 的输出拆分成段落，在段落之间插入 React 组件。或者使用 xterm.js 仅处理 SSH 数据流，AI 相关 UI 作为覆盖层渲染在终端上方的对应位置。

### 5.4 WebSocket 消息协议

```typescript
// === 客户端 → 服务端 ===

// 用户在提示符输入内容（命令或自然语言统一发送，由服务端分类）
{ type: 'input', payload: { text: string } }

// 用户确认执行命令
{ type: 'command_confirm', payload: { commandId: string } }

// 用户拒绝命令
{ type: 'command_reject', payload: { commandId: string } }

// 用户修改命令
{ type: 'command_modify', payload: { commandId: string, newCommand: string } }

// 确认高危命令
{ type: 'danger_confirm', payload: { commandId: string } }

// 终端尺寸变化
{ type: 'resize', payload: { rows: number, cols: number } }


// === 服务端 → 客户端 ===

// SSH 终端原始输出
{ type: 'terminal_output', payload: { data: string } }

// AI 开始思考
{ type: 'ai_thinking', payload: {} }

// AI 流式文本回复 (增量)
{ type: 'ai_reply_chunk', payload: { text: string } }

// AI 回复结束
{ type: 'ai_reply_end', payload: {} }

// 命令确认卡片
{ type: 'command_card', payload: {
  commandId: string,
  command: string,
  risk: 'low' | 'normal' | 'high',
  autoApproved: boolean  // low 风险自动批准
}}

// 命令执行完毕
{ type: 'command_done', payload: { commandId: string, exitCode: number } }

// 错误
{ type: 'error', payload: { message: string } }
```

### 5.5 AI Agent Prompt 设计

```
你是一个 Linux 运维 AI 助手，直接嵌入在用户的 SSH 终端中。

用户在 shell 提示符处输入了自然语言，你需要：
1. 理解用户意图
2. 给出简洁的中文分析/说明
3. 规划需要执行的 shell 命令

输出格式要求：
- 文字说明部分直接用 Markdown 输出
- 需要执行的命令使用以下 XML 标签包裹：
  <command risk="low">ls -la</command>
  <command risk="normal">cd /etc && cat nginx.conf</command>
  <command risk="high">sudo rm -rf /var/log/old/*</command>
- 一次只输出一条命令，等待执行结果后再规划下一步
- 多步骤任务按顺序逐步进行

risk 等级判断：
- low: 纯读取、无副作用（ls, cat, pwd, whoami, df, free, top 等）
- normal: 有副作用但风险可控（mkdir, touch, cp, git clone, cd, npm install 等）
- high: 可能造成不可逆损害（rm -rf, sudo, dd, mkfs, chmod 递归, kill, 
  操作 /etc /boot /sys 等系统目录, 格式化, 写磁盘等）

当前终端环境信息：
- 用户: {username}
- 主机: {hostname}
- 当前目录: {cwd}
- 操作系统: {os_info}
```

## 六、高危命令识别规则

| 规则 | 示例 | 风险等级 |
|------|------|---------|
| `sudo` 开头 | `sudo rm -rf /tmp/*` | high |
| `rm` 带 `-rf` 或 `-r` | `rm -rf ./old` | high |
| 操作系统关键目录 | `/etc`, `/boot`, `/sys`, `/proc` | high |
| `dd` 写操作 | `dd if=... of=/dev/sda` | high |
| `mkfs` 格式化 | `mkfs.ext4 /dev/sdb1` | high |
| `chmod`/`chown` 递归 | `chmod -R 777 /` | high |
| `kill`/`killall` | `kill -9 1` | high |
| `systemctl stop/disable` | `systemctl stop sshd` | high |
| `iptables` 修改 | `iptables -F` | high |
| `reboot`/`shutdown` | `reboot` | high |
| `curl \| sh` 管道执行 | `curl xxx \| bash` | high |
| `git clone`, `npm install` | `git clone xxx` | normal |
| `mkdir`, `touch`, `cp` | `mkdir /tmp/test` | normal |
| `ls`, `cat`, `pwd`, `df` | `ls -la` | low |
